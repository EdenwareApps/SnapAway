# Feature Request: Signed License Token for Offline Validation

## Context
At the moment, the SnapAway client stores a `license.hash` in the local license file and validates it by recomputing the hash from the provided license key.

This approach is weak because any attacker who can edit the local license file can also update `license.hash` to match the tampered key, allowing forged licenses to be accepted.

## Proposal
Introduce a **cryptographic signature** (e.g., HMAC) on the license data that is generated by the license server and verified by the client.

### Desired behavior
1. Server issues a license response that includes a `signature` field (e.g., HMAC-SHA256).
2. Client stores the license key + machineId + timestamp + signature in the license file.
3. On startup/offline validation, the client recomputes the signature and verifies it matches the stored one.

### Benefits
- Prevents local tampering: modifying the stored license key or timestamp invalidates the signature.
- Does not rely on “stored hash” which is trivially forgeable if the file is writable.
- Enables offline validation without storing a “secret” in the same file that is being validated.

## Server-side changes required
The server must be updated to:
- Hold a **secret signing key** (not exposed to clients).
- Generate a signature when activating/issuing a license.
- Return the signature in the activation response payload.

## Client-side changes (already mostly in place)
The SnapAway client already has an internal `_signLicenseData()` helper which can be repurposed to verify the server-provided signature.

## Suggested format (example)
```json
{
  "success": true,
  "data": {
    "licenseKey": "SNAPAWAY-XJ8C-MJ7T-MEX6",
    "machineId": "...",
    "activatedAt": "...",
    "expiresAt": "...",
    "signature": "<HMAC-SHA256>"
  }
}
```

## Notes
- If the server cannot be modified, the client can still protect against trivial tampering by making the stored signature depend on an internal secret, but that is less robust because the secret can be discovered from the client itself.
- A server-issued signature is strongly preferred for real security.
